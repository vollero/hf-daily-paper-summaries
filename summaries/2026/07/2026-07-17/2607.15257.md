# SearchOS-V1: Towards Robust Open-Domain Information-Seeking Agent Collaboration

> [← Daily summary](../2026-07-17.md) · [Hugging Face](https://huggingface.co/papers/2607.15257) · [arXiv](https://arxiv.org/abs/2607.15257)


## One-sentence takeaway
SearchOS is a multi-agent open-domain search framework that improves structured information-seeking by externalizing search progress, evidence, coverage, failures, scheduling, and reusable search skills into system-managed state.

## Problem and motivation
The paper addresses long-horizon, open-domain information seeking where agents must gather many facts, reconcile sources, and produce structured answers with citations. The authors argue that current single- and multi-agent systems often lose track of progress as histories grow, repeat failed searches, duplicate work, or leave coverage gaps because plans, evidence, and failures remain implicit in conversation context. The stated gap is the lack of explicit, persistent, shared search state for coordinating agents around unresolved entities, attributes, evidence, and failed attempts.

## Approach and contributions
SearchOS formulates information seeking as **relational schema completion with grounded citations**: agents discover entity rows, fill attributes across linked tables, and attach source evidence to each populated value.

Its main contributions are:

1. **Relational search formulation**: open-domain requests become relational schemas with entity rows, attribute values, and citation matrices.
2. **Search-Oriented Context Management**: SOCM maintains Frontier Task, Evidence Graph, Coverage Map, and Failure Memory outside agent conversations.
3. **Pipeline-parallel orchestration**: released worker slots are continuously refilled with unresolved schema gaps instead of waiting for synchronized batches.
4. **Middleware-governed execution**: a Search Tool Middleware Harness injects state, extracts anchored evidence, enforces budgets, and reacts to stalls.
5. **Hierarchical search skills**: reusable orchestrator, strategy, and access skills guide decomposition, search, and source-specific extraction.

The evidence supports the authors’ claim that explicit search state, coverage-aware dispatch, middleware control, and skills improve measured performance and efficiency on the reported benchmarks. It does not establish generality beyond the tested WideSearch/GISA settings, nor isolate every individual skill layer because the skill ablation disables all skill layers together.

## Limitations and validity risks
The authors explicitly note that in open-set tables, coverage measures known rows rather than exhaustive enumeration, so 100% known-cell coverage does not guarantee complete row discovery. Appendix D reinforces this with a case where scope auditing is needed after apparent saturation.

Analyst inference: The main results are reported as Max@3 rather than mean performance, which may overstate typical single-run behavior. The supplied text does not provide full Table 2 numeric values for every baseline, nor statistical uncertainty for the main benchmark results. The skill ablation measures the joint effect of all skill layers, so it does not identify which layer contributes most. Middleware evidence in Figure 5 is described as representative trajectory analysis, not an aggregate causal estimate.

## Open research questions
1. How well does SearchOS generalize beyond WideSearch and GISA?
   This remains open because the paper evaluates only those two benchmarks. A next experiment would test the same system on additional open-domain or domain-specific information-seeking benchmarks.

2. Which SOCM component contributes most to performance?
   The paper defines Frontier Task, Evidence Graph, Coverage Map, and Failure Memory, but the supplied ablations do not isolate each memory component. A next experiment would remove or freeze each component separately.

3. Which skill layer matters most?
   The skill ablation disables hierarchical skills together, leaving orchestrator, strategy, and access skill effects entangled. A next experiment would ablate each layer independently.

4. How robust is SearchOS under lower budgets or weaker models?
   The reported configuration uses GLM-5, Qwen3.5-35B-A3B, 8 parallel sub-agents, and defined time/search limits. A next experiment would sweep model size, concurrency, search budget, and wall-clock budget.
