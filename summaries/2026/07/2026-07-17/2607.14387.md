# Chat2Scenic: An Iterative RAG-Based Framework for Scenario Generation in Autonomous Driving

> [← Daily summary](../2026-07-17.md) · [Relationship graph](relationships.md) · [Hugging Face](https://huggingface.co/papers/2607.14387) · [arXiv](https://arxiv.org/abs/2607.14387)


## One-sentence takeaway

Chat2Scenic iteratively generates regulation-grounded Scenic components with retrieved code examples, achieving 76.42% compilation success and 58.17% end-to-end framework accuracy on a new 123-scenario benchmark using Gemini-3-Flash.

## Problem and motivation

Autonomous-driving systems require diverse, regulation-compliant simulation scenarios, but translating complex regulatory descriptions into executable domain-specific-language programs remains difficult. The authors identify a trade-off: retrieval-and-assembly systems compile more reliably but are constrained by stored components, while full-script generation is more flexible but has low compilation success. Existing evaluations also emphasize simplified descriptions and provide limited assessment of both compilation and semantic fidelity (Sections I, I-B).

## Approach and contributions

- **Iterative generation:** Chat2Scenic parses a natural-language request into global configuration, spatial relations, ego behavior, objects, and restrictions, then generates their Scenic code sequentially while accumulating prior code as context (Section II; Algorithm 1).
- **Dual retrieval:** A semantic code retriever supplies the three most similar component-level Scenic snippets; a hybrid documentation retriever combines embedding search, BM25, and reciprocal-rank fusion (Section II-B).
- **Prompting and interaction:** Specialized generators combine contextual prompting, chain-of-thought, fixed examples, and retrieval-augmented examples. A stateful chatbot supports iterative refinement (Sections II-A and II-C).
- **Evaluation assets:** The authors introduce a 123-scenario benchmark drawn from CARLA Leaderboard, NHTSA crash/pre-crash sources, and UN R152/R157/R171, plus compilation, semantic-layer, and framework-level metrics (Section III-A; Table II).
- **Open implementation:** The paper reports releasing Chat2Scenic’s code publicly (Abstract; Section I-C).

The evidence supports better compilation and end-to-end semantic accuracy than the two re-evaluated public baselines under the paper’s shared benchmark and Gemini-3-Flash setting. The ablation also supports the value of the combined prompting recipe, particularly retrieved code examples.

It does not establish correctness in real vehicles, detection of actual ADS failures, generalization beyond Scenic/CARLA, or superiority over methods without available implementations. Semantic quality was human-scored only after successful compilation, and FA incorporates compilation by multiplying CSR and SQ.

## Limitations and validity risks

- **Author-stated trade-off:** Iterative generation is slow: the best configuration averages 222.11 seconds and 52,339 tokens per scenario. The authors consider this acceptable for offline use (Section III-B2; Table IV).
- **Author-stated model weakness:** Evaluated open-source models produce near-zero compilation success, limiting practical deployment with smaller local models (Section III-B2; Table IV).
- **Author-stated retrieval result:** Documentation retrieval increases cost and lowers performance in the tested configuration (Section III-B1; Table III).
- **Analyst inference — narrow execution environment:** All reported tests use CARLA 0.9.15 and Scenic 3.1.0 on one hardware configuration, so portability to other DSLs, simulators, or versions is untested (Section III-A).
- **Analyst inference — limited baseline coverage:** Only ChatScene and NL2Scenic are quantitatively compared because public implementations were available; the evidence does not cover every method discussed in Table I.
- **Analyst inference — human-evaluation uncertainty:** The paper describes a “human evaluator” but does not report evaluator count, agreement, blinding, scoring scale details, or confidence intervals (Section III-A3).
- **Analyst inference — statistical uncertainty:** No repeated runs, variance estimates, significance tests, or sensitivity to stochastic generation are reported in the supplied text.
- **Analyst inference — possible retrieval contamination is unresolved:** The code database uses official Scenic scenarios, while the benchmark includes CARLA Leaderboard scenarios; the text does not explain overlap controls or train/test-style separation (Sections II-B1 and III-A1).

## Open research questions

1. **Can compiler or simulator feedback close the remaining execution gap?** CSR remains 76.42%, leaving nearly one-quarter of outputs uncompilable. A next experiment could compare one-pass generation with iterative compiler-error repair and simulation-in-the-loop feedback.

2. **Does the method generalize beyond Scenic and CARLA?** The evaluation covers one DSL/simulator pair. Porting the logical schema and retrievers to another scenario language and simulator, then testing equivalent specifications, would assess architectural generality.

3. **Why does documentation retrieval degrade generation?** C12 is slower and less accurate than C11, but the causal mechanism is not isolated. Experiments varying document chunking, retrieval depth, fusion, and placement in the prompt could test whether irrelevant context or context length causes the decline.

4. **How reliable is semantic evaluation?** Human scoring lacks reported agreement and uncertainty. A blinded multi-rater evaluation with a published rubric, inter-rater agreement, and per-source confidence intervals would strengthen validity.
