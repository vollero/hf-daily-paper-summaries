# Tencent WorkBuddy Bench: A Multi-Domain Coding-Agent Benchmark with Contamination-Resistant Task Construction

> [← Daily summary](../2026-07-24.md) · [Relationship graph](relationships.md) · [Hugging Face](https://huggingface.co/papers/2607.20911) · [arXiv](https://arxiv.org/abs/2607.20911)


## One-sentence takeaway

Tencent WorkBuddy Bench openly evaluates coding agents on 260 Code, Web, Office, and Security tasks, using rewritten real-work scenarios, isolated task-specific verification, and dual-harness testing to expose substantial domain and harness sensitivity.

## Problem and motivation

The authors argue that existing coding-agent benchmarks trade off realism, contamination resistance, and auditability: public static benchmarks expose searchable prompts and solutions, while production-derived benchmarks may reflect real usage but remain closed and difficult to audit. They also identify a scope gap: coding agents in organizations increasingly handle repositories, web artifacts, office files, and security workflows rather than bug fixing alone.

The benchmark therefore targets realistic, underspecified workspace tasks while attempting to reduce prompt-level contamination without keeping tasks secret. Its central design goal is an open, reproducible suite whose task distribution is informed by aggregate usage patterns and whose prompts cannot be recovered by searching the originating issue, commit, or pull request (Sections 1–2).

## Approach and contributions

1. **Four-domain benchmark:** The suite contains 80 Code, 70 Web, 50 Office, and 60 Security tasks, all represented as natural-language requests over sandboxed workspaces but evaluated with domain-specific verifiers (Sections 3.1–3.4).

2. **Contamination-resistant construction:** Tasks are reverse-engineered from commits, pull requests, historical CVEs, or business scenarios and rewritten as colloquial, role-based, deliberately underspecified requests. No raw user prompts or sessions are reused; post-release exposure is managed through versioning and optional canaries rather than secrecy (Section 2).

3. **Open, isolated evaluation:** Task directories, environments, tests, scoring code, and reference solutions are released. Evaluation assets remain unavailable during the agent episode and are introduced only afterward (Sections 2.1 and 4).

4. **Track-specific verification:** Code uses hidden tests; Web combines deterministic, LLM/VLM, and interactive agent checks; Office combines deterministic rules with evidence-grounded LLM judgments; Security uses deterministic task programs and anti-cheat tests (Section 4).

5. **Dual-harness leaderboard:** Seven models are evaluated under CodeBuddy Code and Claude Code, using think mode and three runs per track/harness configuration, revealing that rankings depend on both domain and harness (Section 5).

These results establish performance differences within this benchmark’s specific task snapshots, harness builds, endpoints, and scoring systems. They also demonstrate that harness choice can alter absolute scores and rankings. They do not establish general superiority across all agent tasks, languages, serving stacks, or future benchmark versions. Cross-track score comparisons are particularly limited because the four tracks use different scoring instruments and the authors explicitly prohibit a suite-wide average.

## Limitations and validity risks

**Author-stated limitations:**

- Claude Opus 4.8’s Code result under Claude Code used an extra complete-in-one-pass/do-not-ask instruction, reducing direct comparability with standard-protocol cells (Section 7; Table 6).
- Code is dominated by Python, so its findings may not transfer to other languages or ecosystems (Section 7).
- Open publication permits future training exposure; versioning mitigates but cannot eliminate post-release contamination (Sections 2 and 7).
- Web and Office depend partly on model judges, whose style or model-family biases were not separately quantified (Sections 4 and 7).
- Scores depend on particular harness builds and serving arrangements; HY-3 was first-party served, while other models used third-party endpoints (Sections 4 and 7).
- Office is text-first and does not test OCR, visual layout judgment, VLM-based perception, or native desktop GUI interaction (Section 7).
- Cross-track scores are not comparable because each track uses a different verifier (Sections 1–2).

**Analyst inferences:**

- The contamination claim is intentionally narrow: rewriting blocks direct prompt lookup but cannot rule out prior exposure to source code, commits, CVE analyses, or released benchmark artifacts. This follows directly from the residual exposures acknowledged in Section 2.
- The absence of a quantified judge-bias study makes it difficult to determine how much Web and Office rankings reflect evaluator preferences rather than task quality.
- Aggregate usage taxonomies are not released or characterized in sufficient detail here to independently verify how closely task distributions match real organizational demand.
- Three-run averages reveal some variation but the supplied text does not report confidence intervals or per-task statistical uncertainty.

## Open research questions

1. **How durable is contamination resistance after public release?**  
   The paper acknowledges that versioning only mitigates future exposure. A next experiment could track score changes across benchmark versions and test models for canary or task-specific memorization.

2. **How much do model judges alter Web and Office rankings?**  
   Judge bias is acknowledged but not quantified. A follow-up could rescore the same artifacts with multiple independent judges and human adjudication, then measure agreement and ranking stability.

3. **Do the Code findings generalize beyond Python?**  
   Cross-language coverage is explicitly limited. A matched extension could translate equivalent repository tasks into several language ecosystems while holding task intent and verifier strictness as constant as possible.

4. **Which harness mechanisms cause ranking reversals?**  
   The dual-harness results show large shifts, especially in Security, but do not isolate causal components. A factorial study could vary context management, reasoning passback, tool interfaces, and prompt wrappers independently.

## Confidence and missing information

**Confidence: high** for the benchmark design, task counts, scoring protocols, stated leaderboard leaders, and limitations because the supplied text describes them explicitly.

Material missing information includes the full numeric contents of Tables 1–7, per-run variance or confidence intervals, complete per-task results, exact model-judge identities and calibration results, detailed internal usage-taxonomy distributions, exact harness build identifiers, and external replication evidence.
