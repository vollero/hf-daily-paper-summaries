# Environment-free Synthetic Data Generation for API-Calling Agents

> [← Daily summary](../2026-07-21.md) · [Relationship graph](relationships.md) · [Hugging Face](https://huggingface.co/papers/2607.16900) · [arXiv](https://arxiv.org/abs/2607.16900)


## One-sentence takeaway

ESAT generates stateful, multi-step API-agent training trajectories from API specifications alone, and its synthetic supervision substantially improves 1.7B–27B models on AppWorld and OfficeBench without executing APIs during data generation.

## Problem and motivation

Training API-calling agents requires trajectories containing instructions, reasoning, API calls, observations, state changes, and final answers. Existing synthesis pipelines commonly depend on executable APIs and populated backend databases, making adaptation to new API ecosystems costly and slow (Introduction, pp. 1–2).

The authors target this infrastructure bottleneck: generating useful trajectories without a functional environment, human-written tasks, or seed trajectories. This matters especially for long-horizon workflows involving both retrieval and state-changing calls, where simulated identifiers and state can drift across turns (Related Work, p. 3).

## Approach and contributions

- **ESAT pipeline:** Starting only from API specifications, an LLM generates bucketed tasks, judges and rewrites them, then a teacher agent interacts step-by-step with an LLM API simulator; a trajectory-level judge retains majority-approved solutions (Fig. 1; Secs. 3.1–3.3, pp. 2–5).
- **State-aware API simulation:** Responses condition on the specification, arguments, task, same-app interaction history, virtual-user profile, and current time. Deterministic input checks, schema validation, retry feedback, and semantic judging for read APIs constrain outputs (Sec. 3.2, p. 4; Secs. H.1–H.6, pp. 28–29).
- **Coverage-oriented synthesis:** A 360-bucket grid varies difficulty, read/write mix, reasoning focus, app count, and API count; inverse-frequency sampling emphasizes underused apps and endpoints (Secs. G.1–G.2, p. 25).
- **Empirical contribution:** The authors evaluate environment-free SFT data across eight Qwen models on AppWorld and OfficeBench, including comparisons with real-environment AppWorld trajectories and existing synthetic tool-use datasets (Sec. 4; Tables 1–5).

The results establish that, in these two benchmarks and model families, ESAT supervision improves downstream task completion and can outperform the paper’s real-environment and synthetic-data training comparisons. They do not establish that simulated responses reproduce real backend distributions, that gains transfer to arbitrary API ecosystems, or that environment-free data is universally preferable to executable supervision. Several quality assessments also depend on other LLMs as judges.

## Limitations and validity risks

- **Source truncation:** The supplied paper text explicitly omits 26,540 characters from the middle. Consequently, this digest cannot claim complete coverage, particularly of some prompt listings and example material.
- **Long-response degradation:** Simulator failure reaches 23% for outputs above 2K tokens; the authors attribute this potentially to the chosen simulator model (Sec. 4.4; Fig. 2).
- **Uneven task composition:** ESAT-S52 contains only 3% three-app trajectories, and ESAT-AW7 only 7%, limiting evidence for the most complex cross-app workflows (Figs. 3 and 5).
- **Incomplete OfficeBench API coverage:** Only 14 of 20 APIs appear in successful ESAT-OB trajectories; six Excel/PDF endpoints are absent (Sec. C.3; Fig. 8).
- **High synthesis attrition:** Only 22.09% of initial ESAT-S52 tasks pass the final task judge, and 59.62% of trajectory inputs survive trajectory filtering (Tables 9–10). OfficeBench retains 49.38% of trajectory inputs (Table 13).
- **Author-noted evaluation cost:** Large-scale human assessment of simulated calls was considered prohibitively expensive; simulator validity therefore relies mainly on LLM judging (footnote, p. 9).
- **Analyst inference — judge dependence:** Filtering quality varies sharply by judge, and task generation, simulation, validation, and filtering are predominantly model-mediated. This creates correlated-error and reproducibility risks not resolved by the reported precision and 200-item human study.
- **Analyst inference — write semantics:** Appendix H.5 says semantic judging is applied only to GET responses, while write responses are accepted after structural validation. Schema-valid acknowledgments therefore need not guarantee realistic state mutation.
- **Analyst inference — benchmark tuning risk:** AppWorld train and dev splits are used for checkpoint selection during synthetic-data training (Sec. D, p. 21), so the reported test results are valid benchmark results but do not isolate completely benchmark-agnostic model selection.

## Open research questions

1. **Can write-operation state transitions be validated as reliably as reads?** The simulator semantically judges GET responses but not write responses. A next experiment could execute matched simulated and real write-call sequences, then compare subsequent reads and final state invariants.

2. **How much does explicit interaction history contribute?** The supplied experiments do not isolate full-history simulation from turn-local or stateless simulation. An ablation could compare per-app history, full cross-app history, compressed state, and no history on simulator validity and downstream completion.

3. **Can complex multi-app coverage be improved without lowering yield?** Three-app examples are rare in ESAT-AW7 and ESAT-S52. A controlled experiment could oversample three-app buckets and measure acceptance rate, simulator failures, trajectory diversity, and 3-app OfficeBench performance.

4. **How robust is ESAT across simulator and judge models?** Only a limited judge ablation is reported, and simulator results center on GLM-5.1-FP8. A factorial experiment varying generator, simulator, teacher, and judge models could separate model-specific effects from pipeline-level benefits.
