# FVAttn: Adaptive Sparse Attention with Runtime Load Balancing for Video Generation

> [← Daily summary](../2026-07-23.md) · [Relationship graph](relationships.md) · [Hugging Face](https://huggingface.co/papers/2607.16190) · [arXiv](https://arxiv.org/abs/2607.16190)


## One-sentence takeaway

FVAttn accelerates multi-GPU video diffusion inference by repairing the runtime load imbalance created by adaptive Top-p sparse attention and using residual GPU slack to compute additional high-value attention blocks.

## Problem and motivation

Video Diffusion Transformers process long spatiotemporal token sequences, making quadratic self-attention a major inference bottleneck; the authors report that attention occupies 74.1% of the runtime for one 5-second, 720p Wan2.2-14B I2V setting (§1).

Training-free Top-p sparse attention reduces computation while adapting to each attention distribution, but it assigns different workloads to different heads. Under Ulysses-style sequence parallelism, uneven head placement produces rank-level stragglers and synchronization idle time. Predictive or history-based layouts are unreliable in the targeted four-step regime: adjacent-step workload changes reach 97% per head and 44% per rank (Figure 2). Conversely, globally repartitioning after observing the current mask would require costly additional communication (§1).

## Approach and contributions

- **Adaptive sparse-routing frontend:** FVAttn estimates block importance from pooled queries and keys, organizes spatial blocks with a Hilbert curve, applies a Top-p cumulative-mass threshold, and enforces a Top-k minimum as protection against excessive sparsification (§§2.1, 3.1).

- **Runtime Load Balancing (RLB):** After materializing the current mask, RLB measures actual per-head workloads and searches a constrained migration schedule. In the default eight-GPU configuration, each rank moves at most one local head through peer-to-peer communication, targeting the current critical path without a global repartition (§3.2).

- **Slack-Aware Sparse Augmentation (SASA):** Non-critical ranks spend a discounted portion of their measured residual slack on additional not-yet-selected blocks, following importance rankings already produced by routing. This is intended to improve mask coverage without creating a new straggler (§3.3).

- **Overlapped runtime:** CPU scheduling overlaps GPU value quantization, while asynchronous migration and augmentation overlap useful computation. The authors also remove synchronization and kernel-management inefficiencies (§3.4).

These results support the authors’ claim that current-mask load repair reduces straggler latency and that slack reuse can improve dense-output similarity with little additional critical-path time in the tested configurations. They do not establish equivalent benefits on other hardware interconnects, other parallelism schemes, training workloads, longer denoising schedules, or model families beyond the three Wan workloads. VBench differences are also not accompanied by uncertainty estimates or significance tests in the supplied text.

## Limitations and validity risks

- **Author-stated:** Benefits diminish for short sequences or workloads with weak attention sparsity, including many image-generation tasks (§5).

- **Author-stated:** Performance depends on the communication-to-computation ratio. PCIe-connected systems may obtain less benefit than the tested NVLink H20 server and may require recalibrating migration and SASA parameters (§5).

- **Author-stated:** Validation focuses on Ulysses-style parallelism and few-step video DiT inference; Ring Attention, USP, training-time attention, and broader hardware topologies remain untested (§5).

- **Analyst inference:** Hardware generality is limited because all reported experiments use one 8× H20 server with CUDA 13.1 (§4.1), although Figure 8 reportedly studies multiple GPU counts on that environment.

- **Analyst inference:** Model diversity is narrow: all three workloads are 14B Wan variants using the same LightX2V four-step distilled configuration (§4.1).

- **Analyst inference:** Animate quality evidence is particularly limited because it uses an internal set of only 20 cases, unlike the 946-case T2V and 1,118-case I2V VBench sets (§4.1).

- **Analyst inference:** The supplied text provides fixed thresholds and budgets but no sensitivity results for \(k_1\), \(k_2\), \(n_1\), or \(n_2\), leaving calibration robustness unclear (§§3.3, 4.1).

- **Analyst inference:** No public implementation, complete kernel specification, variance across runs, or statistical significance analysis is established by the supplied text.

## Open research questions

1. **How robust are RLB and SASA across interconnects and GPU generations?** The paper tests only an NVLink-connected H20 server. A next experiment could compare H20/NVLink with PCIe-only and alternative accelerator clusters while sweeping migration budgets and SASA thresholds.

2. **Do the gains transfer to other sequence-parallel layouts?** Ring Attention and USP are explicitly outside the validated scope. A controlled experiment could implement the same realized-workload repair under each layout and compare migration cost, imbalance, and attention latency.

3. **How sensitive is performance to the runtime thresholds?** The paper reports one principal configuration without a supplied sensitivity ablation. A factorial sweep over \(k_1\), \(k_2\), \(n_1\), and \(n_2\) could measure quality, latency, and new-straggler frequency across workloads.

4. **Does runtime repair remain effective beyond four-step Wan models?** Step-to-step variability motivates the approach, but broader denoising schedules and model families are not evaluated. A next experiment could test distilled and non-distilled video DiTs at several step counts using identical resolutions and hardware.

## Confidence and missing information

**Confidence: high** for the method description and reported results because the supplied text includes the full design, experimental setup, ablations, and limitations.

Material missing information includes public code and artifact availability, implementation details sufficient for independent kernel reproduction, run-to-run variance, confidence intervals or significance tests, threshold-sensitivity studies, energy and memory measurements, true end-to-end latency including encoders and VAE, and evaluations on non-Wan models, non-H20 hardware, PCIe-only systems, or sequence-parallel methods other than Ulysses.
