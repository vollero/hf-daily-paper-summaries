# KnowAct-GUIClaw: Know Deeply, Act Perfectly, Personal GUI Assistant with Self-Evolving Memory and Skill

> [← Daily summary](../2026-07-16.md) · [Hugging Face](https://huggingface.co/papers/2607.12625) · [arXiv](https://arxiv.org/abs/2607.12625)


## One-sentence takeaway
KnowAct-GUIClaw augments OpenClaw-style personal agents with host-guided routing, persistent experience memory, typed cross-app data transfer, and state-validated reusable GUI skills, improving mobile GUI task success and efficiency in the authors’ reported evaluations.

## Problem and motivation
The paper addresses GUI automation for personal assistants that must operate across apps, devices, and long-horizon workflows when APIs are unavailable. The authors identify two main gaps in existing OpenClaw-style agents: weak cross-platform GUI interaction support and no built-in self-evolution from prior execution experience. They argue that standalone GUI agents are fragile for personal-assistant tasks because they lose intermediate values, suffer from partial observability, discard useful trajectories, and miss faster non-visual shortcuts such as deeplinks, intents, and reusable action sequences (Introduction).

## Approach and contributions
The method is a Know–Route–Act–Reflect framework. A host agent manages user context, memory, routing, external tools, and decomposition, while a GUI executor handles screen perception, normalized actions, device backends, state validation, and trajectory logging.

1. Two-tier host–executor collaboration: the host selectively delegates GUI subtasks and can resume, reroute, or solve eligible information subtasks itself (Sections 4.1–4.2).
2. Memory-grounded routing and typed information transfer: cross-app workflows are decomposed into app-scoped subtasks with declared inputs and outputs passed through a blackboard (Section 4.3; Figure 4).
3. Knowledge- and skill-augmented GUI execution: repeated GUI behavior, predefined action sequences, deeplinks, and intents become state-validated skills or shortcuts in a hybrid action space (Section 4.4; Appendix B/D).
4. Reflection-based self-evolution: trajectories are distilled into textual experience memory or repaired/extracted skills after execution (Section 4.5).

The evidence supports the authors’ claim that their architecture improves success rate and can reduce GUI-step/token cost in the tested MobileWorld ablations. It does not establish universal model-independent transfer, broad production robustness, or fully benchmark-standard cross-platform performance, because transfer is shown only for a Kimi-to-Qwen direction and the HarmonyOS/Windows checks use smaller, partly manual protocols.

## Limitations and validity risks
The authors state that future work should reduce rigid sequential handoffs by creating a unified joint planner that weighs knowledge retrieval, tool calls, and GUI actions in one decision cycle (Conclusion). They also report concrete failure modes: HarmonyOS wheel pickers, payment-page semantic mismatch, system-UI swipe mismatch, Windows emoji grounding, short-lived notification timing, continuous geometric control, deep app-specific controls, and off-viewport targets (Section 5.6).

Analyst inference: AndroidDaily comparability is limited because it is evaluated on iOS devices, includes unavailable entries, and uses LLM/human judging rather than a native deterministic evaluator (Sections 5.1 and 5.4). Cross-platform evidence is useful but narrow because HarmonyOS uses selected MobileWorld-derived tasks and Windows uses a manually designed set (Section 5.6). Reproducibility is partially supported by logs, but the supplied text does not provide enough detail to independently recreate skill stores, memory contents, prompts in full, or all device states.

## Open research questions
1. Can a unified planner outperform the sequential Know–Route–Act–Reflect pipeline? This remains open because the authors identify handoff overhead as future work; a next experiment would compare the current staged pipeline with a joint planner on MobileWorld token cost and SR.
2. How general is skill and memory transfer across model families and platforms? The paper shows Kimi-to-Qwen transfer only; a next experiment would test bidirectional and multi-model transfer across MobileWorld, AndroidDaily, HarmonyOS, and Windows.
3. Do state-validated skills remain reliable under app UI updates? The paper discusses skill repair and stale target refresh but does not quantify long-term drift; a next experiment would rerun stored skills across app versions and measure repair success.
4. How much does each memory type contribute separately? The ablations combine host, router, and experience memory; a next experiment would isolate routing memory, user/profile memory, success lessons, failure lessons, and executable skills.
