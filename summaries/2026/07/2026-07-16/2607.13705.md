# AgentCompass: A Unified Evaluation Infrastructure for Agent Capabilities

> [← Daily summary](../2026-07-16.md) · [Relationship graph](relationships.md) · [Hugging Face](https://huggingface.co/papers/2607.13705) · [arXiv](https://arxiv.org/abs/2607.13705)


## One-sentence takeaway
AgentCompass is an open-source infrastructure that decouples benchmarks, harnesses, models, and environments to make LLM-agent evaluation more reproducible, extensible, and diagnosable beyond scalar benchmark scores.

## Problem and motivation
The paper addresses fragmented LLM-agent evaluation pipelines, where benchmarks often require separate environment setup, data formats, scoring logic, and agent integration. The authors argue this causes redundant engineering and weakens reproducibility because baseline implementations and execution protocols differ across systems. The gap they identify is the lack of a unified infrastructure that supports interactive agent workflows across diverse capabilities while preserving flexible component reuse. Evidence: Abstract; Section 1.

## Approach and contributions
1. **Modular evaluation architecture:** AgentCompass separates evaluation into Benchmark, Harness, Environment, and Model components, connected through stable protocols and declarative run specifications. Evidence: Sections 3.1–3.2; Figure 2.

2. **Composable agent evaluation:** The framework supports benchmark × harness × environment configurations, allowing different agents, interaction protocols, and execution backends to be combined without rewriting benchmark logic. Evidence: Sections 3.1, 3.6.

3. **Fault-tolerant asynchronous runtime:** It uses asyncio-based task dispatch, concurrency limits, incremental result persistence, and resumable execution. Evidence: Section 3.3.

4. **Trajectory analysis tools:** AgentCompass records full interaction trajectories and provides analyzers for failures such as output truncation, latency spikes, repeated generation loops, and suspected reward hacking. Evidence: Sections 3.4, 4.3.

5. **Built-in benchmark and harness support:** The authors claim native support for over 20 benchmarks across five capability dimensions and multiple representative harnesses. Evidence: Abstract; Section 3.6; Tables 1–2.

The evidence supports the authors’ claim that unified infrastructure can expose harness sensitivity, reproducibility gaps, and trajectory-level behavioral differences. It does not fully establish numerical superiority of AgentCompass over other infrastructures, because the supplied text does not include complete table values, ablations against alternative frameworks, or direct measurements of engineering effort saved.

## Limitations and validity risks
The authors state that suspected reward hacking is defined behaviorally rather than evidentially, so flagged actions do not prove causal manipulation of benchmark outcomes. Evidence: Section 4.3 RQ2.

The supplied text references several important tables and figures but does not include their numerical contents, limiting independent assessment of benchmark scores, percentages, and step statistics. Analyst inference based on missing table contents.

The comparisons to official baselines may be affected by harness versions and benchmark-specific adaptations, which the authors explicitly acknowledge. Evidence: Section 4.2.

The evaluation uses many benchmark-specific workflows, judges, timeouts, and compatibility fixes, so results may depend on configuration choices despite the unified framework. Evidence: Appendix A.

## Open research questions
1. How much does each architectural component independently improve reproducibility?
This remains open because the paper describes modularity but does not provide ablations comparing coupled versus decoupled pipelines. A next experiment would compare result variance and setup effort across identical benchmarks implemented inside and outside AgentCompass.

2. How reliable are the trajectory analyzers across benchmark families?
The paper shows example analyses, but the supplied text does not report analyzer precision, recall, or human agreement. A next experiment would manually label trajectory failures and evaluate analyzer accuracy.

3. Can suspected reward-hacking detection distinguish harmful behavior from benign repair strategies?
The paper explicitly defines suspected hacking behaviorally, not causally. A next experiment would test whether flagged behaviors actually change benchmark outcomes under controlled environment restrictions.

4. How stable are harness-sensitive rankings across versions?
The authors note that harness versions and adaptations may affect results. A next experiment would rerun the same models across multiple harness versions and report rank stability.
