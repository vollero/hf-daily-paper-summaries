# PalmClaw: A Native On-Device Agent Framework for Mobile Phones

> [← Daily summary](../2026-07-16.md) · [Hugging Face](https://huggingface.co/papers/2607.13027) · [arXiv](https://arxiv.org/abs/2607.13027)


## One-sentence takeaway
PalmClaw is an open-source mobile-native LLM agent framework that runs agent state and tool execution on Android phones, using explicit device tools to improve task success, latency, setup burden, and execution boundaries over GUI-centered mobile-agent baselines.

## Problem and motivation
The paper addresses how LLM agents can operate directly on mobile phones rather than through desktop/cloud-hosted orchestration or GUI-only phone control. The authors argue this matters because phones are widely available, hold personal data, expose sensors, and host daily-use apps. They identify a gap in existing mobile agents: GUI actions such as tapping and swiping can require long layout-sensitive sequences, do not directly expose device resources, and give broad interface-level access without clear task-specific execution boundaries. Evidence: Abstract; Section 1; Figure 1.

## Approach and contributions
PalmClaw runs sessions, memory, skills, tools, and the agent loop inside a mobile app, while using a remote LLM API for inference. Device tools expose phone capabilities through JSON argument schemas, structured results, permission checks, workspace checks, and user confirmations. Evidence: Sections 3.1–3.3; Figure 2.

Contributions:
1. A mobile-native open-source agent framework hosting agent loop, memory, skills, tools, and session state on the phone. Evidence: Section 1 contributions; Section 3.
2. Device tools for explicit, bounded access to mobile resources and capabilities. Evidence: Sections 3.2–3.3.
3. Evaluation on mobile tasks, deployment/setup burden, and execution-boundary traces. Evidence: Section 4; Tables 1–3; Figure 4.

## Evidence and main results

| Finding | Evidence / setting | Location in paper |
|---|---|---|
| PalmClaw achieved 97.1% success on MobileTask. | 70 mobile tasks; compared with MobileClaw 60.0%, ClawMobile 77.1%, ApkClaw 87.1%. | Table 1; Section 4.2 |
| PalmClaw reduced average completion time to 17.7s. | MobileTask average time per task; strongest baseline by success was ApkClaw at 348.8s. | Table 1; Section 4.2 |
| PalmClaw used fewer actions and tokens. | 2.8 actions and 50.4K tokens versus ApkClaw 103.9 actions and 2.06M tokens, ClawMobile 51.4 actions and 466.4K tokens, MobileClaw 94.3 actions and 1.30M tokens. | Table 1 |
| PalmClaw reached 36.85% AssistantBench accuracy. | 19-task selected development subset; baselines reported 5.26%, 10.49%, and 25.79%. | Table 1; Section 4.1 |
| PalmClaw required fewer setup steps and less setup time. | 2 setup steps and about 2 minutes; no separate computer, CLI, or bridge required. | Table 2; Section 4.3 |
| Execution boundaries are demonstrated through permission and workspace cases. | Calendar permission triggers permission request; external file path triggers settings guidance for all-files access. | Section 4.4; Figure 4; Table 3 |

The evidence supports the authors’ claim that PalmClaw is faster and more successful than the tested open-source baselines on their selected MobileTask setup, and that it has lower visible setup burden in their manual deployment comparison. It does not establish general superiority across all mobile agents, all phones, all LLM backbones, private user data workflows, or broader AssistantBench settings.

## Limitations and validity risks
Author-stated or directly supported limits include reliance on a remote LLM API, meaning prompts, tool results, or task context may be sent to provider-side systems; the ethics statement says this data flow should be clear to users. The AssistantBench evaluation uses only 19 selected development tasks because the local test split lacks gold answers and some dev tasks were unstable. Evidence: Sections 3.1, 4.1, Ethics Statement, Appendix A.2.

Analyst inference: the evaluation is narrow because mobile experiments use one Android phone model and Android 13, and the text does not report variation across devices, OS versions, users, or LLM backbones. The paper reports trace-based execution-boundary cases but does not provide a quantitative safety or security evaluation. It also does not establish whether MobileTask favors device-tool designs despite the authors’ stated filtering rules.

## Open research questions
1. How robust is PalmClaw across different Android devices, OS versions, and permission implementations? This remains open because experiments report one Xiaomi Redmi Android 13 setup. A next experiment would run MobileTask across multiple phone models and Android versions.

2. How much of the gain comes from the device-tool action space versus other framework choices? This remains open because the supplied text does not report ablations. A next experiment would compare PalmClaw with and without device tools under the same agent loop and LLM.

3. How reliable are execution boundaries under adversarial or ambiguous user requests? This remains open because the paper provides trace cases, not a systematic boundary stress test. A next experiment would build a benchmark of permission, workspace, confirmation, and unsupported-action challenges.

4. Does PalmClaw remain effective with local on-device LLM inference? This remains open because inference is provided by a remote LLM API. A next experiment would compare remote and local models on the same tool suite and task set.
