# Generative Compilation: On-the-Fly Compiler Feedback as AI Generates Code

> [← Daily summary](../2026-07-16.md) · [Relationship graph](relationships.md) · [Hugging Face](https://huggingface.co/papers/2607.13921) · [arXiv](https://arxiv.org/abs/2607.13921)


## One-sentence takeaway

Generative compilation seals incomplete Rust programs into compilable forms, enabling standard compilers to provide actionable feedback during black-box LLM generation and reducing compiler errors versus post-generation feedback from 20.7% to 13.1% on average.

## Problem and motivation

Compiler feedback ordinarily arrives only after an LLM finishes generating a file, allowing early mistakes to cascade and producing large diagnostic bundles. Constrained decoding can intervene earlier, but semantic constraints require substantial language-specific reimplementation, white-box access to sampling, and provide no explanatory diagnostics (§§1–2).

The authors address the gap between these approaches: obtaining faithful, compiler-style diagnostics on partial programs while retaining existing compiler infrastructure and compatibility with streaming black-box models.

## Approach and contributions

- **Generative compilation:** A concurrent feedback loop checks streamed program prefixes and restarts generation with mapped compiler diagnostics when a prefix is rejected (§3; Fig. 6).
- **Sealors:** A language-specific transformation closes partial syntax and inserts well-typed placeholders so an ordinary compiler can inspect incomplete programs (§3.1).
- **Formal foundation:** Completeness prevents rejection of prefixes with valid completions; soundness identifies dead-end prefixes. These properties lift from a sealor to its induced generative compiler (Theorem 3.2).
- **Mechanized calculus:** The Featherweight Rust sealor is globally complete and sound at statement boundaries, with the language, sealor, and proofs mechanized in Lean (§§4–5; Theorems 5.3 and 5.7).
- **Practical Rust system:** The first claimed partial-program checker for real Rust preserves generated code, inserts `holediv()` and `holeval()`, selectively suppresses future-dependent errors, maps diagnostics back to original prefixes, and delegates semantic checks to `rustc` (§6).
- **Released assets:** The authors report publicly releasing the proof mechanization, implementation, benchmarks, and evaluation results (p. 1 footnote).

## Evidence and main results

| Finding | Evidence / setting | Location in paper |
|---|---|---|
| GC reduced average compiler error rate from 20.7% with post-generation compilation (PC) to 13.1%. | Seven models × two Rust tasks; two samples per model-task; GC was better in 13 configurations and tied in one, with 9 improvements statistically significant at α=5%. | §7.2; Table 1 |
| GC achieved the highest functional correctness in 11 of 14 model-task configurations. | Largest reported gains were PC 53.3% → GC 71.7% for GLM 5.2 on UpdatedAPI and 39.9% → 53.9% for Kimi K2.7 on Translation. | §7.2; Table 1 |
| GC lowered feedback overhead relative to PC. | Average overhead over direct LLM generation was 135 seconds (+170%) for GC versus 233 seconds (+283%) for PC; Qwen 9B Translation fell from 879 to 357 seconds per sample. | §7.2 |
| Early feedback resolved most evaluated tasks without falling back to PC. | Across tasks, 85.3% completed without switching to post-generation feedback, and 55.4% were solved correctly within the first 10 early-feedback restarts. | §7.3 |
| GC produced smaller diagnostic bundles. | 65% of GC reports contained one or two diagnostics and averaged 5.5, versus 40% and an average of 13.8 for PC. | §7.3; Fig. 12a |
| GC localized errors substantially earlier. | Replay of 961 erroneous LLM files: median delay from the final compiler error’s primary span was 3 lines for GC, 14 for function-level GC, and 89 for PC; GC’s mean was 27.7 lines. | §§7.3; Fig. 12b |
| GC detected errors around one-third of the way through a file. | Mean detection point was 33.3% of file length versus a timing-free lower-bound reference of 32.7%; function-level GC detected at 40.3%, while PC waited for completion. | §7.3; Fig. 12c |
| The FR sealor has formal correctness properties. | It is globally complete for arbitrary strings and sound at statement boundaries; the development is mechanized in Lean. | Theorems 5.3 and 5.7; §§4–5 |
| Restart and token limits were reported as non-binding. | Pooling both datasets, increasing restarts from 15 to 20 changed compilable-output rates from 84.9% to 87.0% for GC and 79.2% to 80.5% for PC; outputs remained below token limits. | §A.5; Fig. 14 |

The evidence supports the authors’ claim that on-the-fly compiler feedback improves compilation success and usually improves test-passing performance over their post-generation-feedback baseline under matched restart budgets. It also directly demonstrates earlier, more focused diagnostics.

It does not establish equivalence to constrained decoding, generalization beyond Rust, or formal fidelity of real-Rust diagnostics. Functional correctness is measured by provided unit tests, not full semantic equivalence or security guarantees.

## Limitations and validity risks

- **Author-stated:** The formal results cover verdicts, not whether each diagnostic accurately identifies a defect in the original prefix; diagnostic faithfulness is observed empirically rather than proved (§8).
- **Author-stated:** Both FR and Rust sealors are hand-written, and real Rust receives informal per-feature arguments rather than a mechanized global proof (§§6, 8).
- **Author-stated:** Invocation frequency, agent-controlled use, white-box integration, and use of diagnostics during training remain unexplored (§8).
- **Author-stated:** No Rust constrained-decoding implementation was available, so the evaluation lacks that comparison (§7.1).
- **Analyst inference:** External validity is narrow because evaluation covers two Rust repository-level task families, including only 20 selected difficult CRUST-Bench instances and 30 constructed UpdatedAPI projects (§§7.1, A.1–A.2).
- **Analyst inference:** UpdatedAPI construction used GPT 5.3 to select packages and create use cases before manual assessment, which may introduce model- or curator-dependent selection effects (§A.1).
- **Analyst inference:** Translation selects the 20 CRUST-Bench tasks that two frontier models did not solve zero-shot, so results should not be read as benchmark-wide averages (§A.2).
- **Analyst inference:** Each model was evaluated only twice per task, limiting precision for stochastic model comparisons (§7.1).
- **Analyst inference:** The Rust implementation depends on patched `rust-analyzer`, a patch to Rust 1.95.0, selective diagnostic suppression, and an arity query through `rustc`; these increase maintenance and reproduction sensitivity (§§6.1–6.2, 7.1, A.3).

## Open research questions

1. **Are projected diagnostics always faithful to the unsealed prefix?**  
   The paper proves verdict properties but not diagnostic correctness (§8). A next experiment could annotate a stratified sample of projected diagnostics for sealing artifacts and compare them with an independently derived earliest-error oracle.

2. **Can sealors be synthesized automatically?**  
   Rust and FR rules were manually designed (§8). A plausible experiment would synthesize rules for a small typed language from its grammar and checker, then test completeness and soundness exhaustively on bounded programs.

3. **How should checking frequency be scheduled?**  
   The system uses a latest-wins concurrent loop, but invocation policy is not optimized (§§3.2, 8). An experiment could compare token-, chunk-, statement-, and adaptive compiler-invocation schedules under matched latency or compute budgets.

4. **Does the benefit transfer beyond Rust and these task families?**  
   Empirical validation is Rust-specific. A next study could implement sealors for another statically typed language and repeat matched evaluations on code completion, repair, and repository-editing tasks.
