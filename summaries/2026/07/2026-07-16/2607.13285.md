# Harness Handbook: Making Evolving Agent Harnesses Readable,Navigable, and Editable

> [← Daily summary](../2026-07-16.md) · [Hugging Face](https://huggingface.co/papers/2607.13285) · [arXiv](https://arxiv.org/abs/2607.13285)


## One-sentence takeaway
Harness Handbook is a behavior-centric repository representation for agent harnesses that helps coding agents localize modification sites and produce better edit plans with fewer planner tokens.

## Problem and motivation
The paper addresses behavior localization: finding all source locations that implement a requested harness behavior before editing. The authors argue this matters because agent harness behavior is distributed across files, functions, stages, state transitions, and tool/API interactions, while modification requests are usually behavioral rather than file-oriented. They identify a gap in existing code search, repository maps, summaries, memory, and long-context methods: these organize around implementation structure, but do not explicitly map requested behaviors to all underlying implementation sites. Evidence: Abstract; Section 1.

## Approach and contributions
The authors introduce Harness Handbook, an automatically synthesized representation that organizes harness code by runtime behavior and links each behavior to source locations. It contains an L1-L3 document tree plus a state-register view for cross-stage state relationships. Evidence: Section 3.1; Figure 1.

They propose a construction pipeline using deterministic static fact extraction plus LLM-assisted behavioral organization and synthesis, supporting function-as-leaf and file-as-leaf modes. Evidence: Section 3.2; Figure 2; Algorithm 2; Appendix A.

They introduce Behavior-Guided Progressive Disclosure, which routes a planner from high-level behavior descriptions through handbook stages, registers, and L3 entries, then verifies candidate locations against the current repository. Evidence: Section 3.3.1; Appendix B.1.

They evaluate the method on modification-planning tasks for two open-source agent harnesses, comparing baseline repository exploration against handbook-assisted planning. Evidence: Section 4; Appendix C.

The evidence supports the authors’ claim that the handbook improves localization-oriented planning under their evaluation setup, while reducing planner-token consumption. It does not establish that final code edits are more correct after execution, that the approach generalizes beyond the two evaluated harnesses, or that the LLM-generated handbook is always accurate under different repositories and models.

## Limitations and validity risks
The authors’ evaluation scores edit plans rather than final executed changes, so downstream implementation correctness is not directly established. Evidence: Section 4.1; Appendix D.3.2.

The benchmark covers two open-source harnesses and 60 requests, which supports within-domain evidence but leaves generalization to other harness architectures, languages, and repository scales uncertain. Evidence: Section 4.1; Appendix C.1-C.2.

The method relies partly on LLM-assisted organization and description revision, so handbook quality may depend on model behavior, prompt design, and construction budgets. The paper describes conservative freezing and coverage records, but does not provide a standalone accuracy audit of generated handbook entries in the supplied text. Evidence: Sections 3.2, 3.3.3; Appendix A-B.

Analyst inference: because plan quality is judged by LLMs and reference plans also come from stronger LLMs, the evaluation may favor plans that align with model-produced localization expectations rather than independently verified human ground truth. The supplied text states the judges and reference plans, but does not report human adjudication. Evidence: Section 4.1; Table 1.

## Open research questions
1. Does handbook-assisted planning improve final patch correctness, not just plan quality?  
Why open: the paper scores plans directly and keeps execution out of scope. A plausible next experiment is to execute both arms’ plans and run task-specific tests or human code review on the resulting diffs.

2. How accurate and stable are generated handbook entries over many repository updates?  
Why open: resynchronization is described, but the supplied results do not quantify handbook drift or locator-freezing rates. A plausible next experiment is a longitudinal benchmark with repeated edits, measuring stale locators, frozen entries, and localization accuracy over time.

3. How much does each component contribute: static analysis, state-register view, L1-L3 hierarchy, and BGPD policy?  
Why open: the supplied experiment compares full handbook assistance against baseline exploration, but does not report ablations. A plausible next experiment is to remove each component and measure localization F1, Wrong, and token use.

4. Does the method transfer to non-agent software systems or smaller libraries?  
Why open: the motivation and evaluation focus on agent harnesses. A plausible next experiment is to apply the representation to other behavior-heavy systems, such as workflow engines or developer tools, using the same request taxonomy.

## Confidence and missing information
Confidence: medium.
