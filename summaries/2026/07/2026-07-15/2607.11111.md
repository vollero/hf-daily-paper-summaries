# Know Before Fix: QA-Driven Repository Knowledge Acquisition for Software Issue Resolution

> [← Daily summary](../2026-07-15.md) · [Relationship graph](relationships.md) · [Hugging Face](https://huggingface.co/papers/2607.11111) · [arXiv](https://arxiv.org/abs/2607.11111)


## One-sentence takeaway
ACQUIRE improves LLM software issue repair by making agents ask and answer targeted repository questions before patching, yielding higher Pass@1 on SWE-bench Verified with modest added cost.

## Problem and motivation
The paper addresses factual repair errors caused by insufficient repository understanding: agents may localize by keywords, miss cross-module behavior, or violate implicit API contracts. The authors argue that existing pre-repair methods are still fix-oriented: they surface suspicious locations or summaries, but do not explicitly identify what repository knowledge the repair agent lacks before editing. This matters because repository-level issue resolution often requires internal behavioral, design, structural, or external-standard knowledge not present in the issue description.

## Approach and contributions
ACQUIRE separates issue resolution into two stages.

1. **Question-driven knowledge acquisition:** a Questioner generates targeted questions using four knowledge categories: Mechanism & Behavior, Design & Usage, Locating & Structure, and Ecosystem & Standards. Independent Answerer agents explore the repository in read-only mode and produce evidence-grounded answers.

2. **Knowledge-informed repair:** a Resolver receives the issue plus the generated QA pairs as static context before beginning normal Mini-SWE-Agent-style repair.

The claimed contributions are: introducing repository-level QA as an internal capability for issue resolution; decoupling knowledge acquisition from patch generation; and showing, through audits and trajectory analysis, that QA knowledge is mostly grounded and changes repair behavior.

The evidence supports the authors’ central claim that structured pre-repair QA can improve repair accuracy and reduce knowledge-intensive search effort on SWE-bench Verified, especially under DeepSeek-V3.2 where detailed behavioral analyses are reported. It does not fully establish generality beyond the tested Python-focused SWE-bench Verified setup, nor does it isolate all implementation details because key prompt templates and some result tables are only partially available in the supplied text.

## Limitations and validity risks
The authors state that the current prompt design and evaluation are tailored to Python repositories, so effectiveness across other languages remains unvalidated. They also state that the four-category question taxonomy may need refinement, and that static pre-generated QA may be less adaptive than dynamic knowledge acquisition during repair.

Analyst inference: the supplied text omits the actual numeric contents of several referenced tables, including full Table I baseline cost/time/Pass@1 rows, Table II ablation details beyond selected values, and Table III question-quality numbers beyond reported deltas. This limits independent comparison of all baselines.

Analyst inference: the LLM-as-a-judge question-quality evaluation is a validity risk, although the authors mitigate it with repeated judgments and pairwise voting. The factual audit is human-reviewed but covers 232 QA pairs, not necessarily every generated QA pair. Regression analysis shows that factually supported QA can still frame the repair incorrectly.

## Open research questions
1. **Can QA-driven repair generalize beyond Python repositories?**  
   This remains open because the authors explicitly say prompts and evaluation are tailored to Python. A plausible next experiment is to run ACQUIRE on multi-language repository repair benchmarks with language-adapted and language-neutral question templates.

2. **Would dynamic QA during repair outperform static pre-injection?**  
   The paper uses static QA context and names dynamic acquisition as future work. A next experiment would compare static N=2 QA against a resolver that can request new QA after failed tests or new hypotheses.

3. **How can the Resolver avoid plausible but wrong QA framing?**  
   This remains open because 5 of 22 Pass→Fail regressions were classified as misleading QA, mostly despite factually supported answers. A next experiment would test verifier or critic modules that rank QA relevance before edits.

4. **What is the optimal question taxonomy?**  
   The four categories help, but the authors say they may need refinement. A next experiment could compare manually designed, learned, and issue-type-adaptive taxonomies under the same repair backbone.
